# Forms

Use this reference for forms, validation, complex inputs, dirty state, submission flows, and field-level errors.

Use the existing form library or framework-native pattern. Keep validation rules close to the schema or server contract when possible. Show field errors near fields, summarize errors for long forms, and preserve entered values after failed submissions.

Handle pending, success, failure, disabled, dirty, touched, and optimistic states deliberately. Prevent duplicate submissions where side effects are not idempotent. Keep file uploads, masked inputs, date pickers, and comboboxes accessible with labels, keyboard support, and clear error messages.
